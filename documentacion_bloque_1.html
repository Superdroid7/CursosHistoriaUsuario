Reporte de Misión: Bloque 1 - Fundación Hexagonal

Agente, aquí tienes el reporte detallado de la primera fase de construcción de nuestro Sistema de Cursos. Hemos establecido la base operativa, mapeado las entidades clave y asegurado los corredores de comunicación.

Analogía de Supervivencia (Resident Evil): Construir una arquitectura hexagonal es como explorar la Mansión Spencer. Primero necesitas el mapa central y las llaves maestras (El Dominio). Si empiezas por abrir las puertas exteriores (Controladores o Base de Datos) sin conocer el mapa interno, los zombis (el código espagueti) te devorarán.

1. Orden de Creación: ¿Por qué este camino?

Hemos seguido una secuencia estricta basada en los principios de Clean Architecture (Arquitectura Hexable/Hexagonal):

  • Primero: El Dominio (Category y Course). Como las hierbas verdes y rojas de Resident Evil, son los elementos vitales que no dependen de nada más. Aquí se define "qué es un curso" sin importar si se guarda en Postgres o en un archivo de texto.
  • Segundo: Puertos de Salida (Interfaces de Repositorio). Estas son las ranuras para insertar las Llaves del Castillo. El dominio dice "necesito guardar un curso", pero no le importa cómo. Solo deja la ranura (la interfaz) lista.
  • Tercero: Casos de Uso (Application). Son nuestras armas. Un caso de uso toma un curso (munición) y lo usa a través de una interfaz.
  • Cuarto: Infraestructura (JPA, Entidades de BD y Controladores). Finalmente construimos los pasillos exteriores. Aquí es donde conectamos la Base de Datos (PostgreSQL) usando variables de entorno (como un maletín de almacenamiento seguro en la nube) y creamos los endpoints (puertas) para que los usuarios puedan entrar.

2. Bloques de Código y su Función

El Dominio Puro (Modelos)

public class Course { private Long id; private String title; // ... }

Objetos limpios de Java (POJOs). No saben nada de Spring Boot ni de bases de datos. Son la verdad absoluta de nuestro negocio.

Los Casos de Uso (GetPublicCoursesUseCase)

Esta clase representa la intención del usuario. Si la historia de usuario dice "El visitante quiere ver todos los cursos", esta clase ejecuta exactamente eso llamando al puerto de base de datos.

La Infraestructura: Adaptadores y Entidades

Analogía de Invocador (League of Legends): El Adaptador de Persistencia (CoursePersistenceAdapter) es como Sylas. Roba la información del Dominio (Poder Mágico) y la traduce o adapta al formato que entiende la Base de Datos (JPA Entity), actuando como un puente entre ambos mundos.

Creamos CourseEntity y CategoryEntity específicamente para la base de datos. El adaptador toma nuestro modelo de dominio puro, lo convierte en Entity, lo guarda en la BD, y cuando lo recuperamos, lo vuelve a convertir a Dominio.

3. El Arsenal: Uso de Anotaciones

A lo largo de este bloque hemos utilizado potentes hechizos (anotaciones) proporcionados por nuestros frameworks (Spring Boot, Lombok, SpringDoc):

  • @Entity y @Table(name="..."): Marcan una clase como una tabla de base de datos. Como ponerle un sello de "Propiedad de Umbrella Corp" a la clase para que Hibernate sepa cómo guardarla.
  • @RestController y @RequestMapping: Abren una puerta de enlace pública (API REST) hacia el mundo exterior, escuchando peticiones HTTP.
  • @Data, @Builder, @RequiredArgsConstructor: Magia de Lombok. Reducen el código repetitivo (getters, setters, constructores) para que nuestro código no parezca un laberinto de Silent Hill. Al usar @RequiredArgsConstructor, Spring inyecta automáticamente las dependencias (como los puertos o repositorios) sin necesidad de escribir constructores manuales o usar @Autowired.
  • @Tag y @Operation: Munición de SpringDoc (Swagger). Documentan automáticamente nuestras rutas. Es como encontrar un mapa del juego pegado en la pared: cualquiera que lea la documentación sabrá exactamente qué hace cada Endpoint.
Para la Base de Datos en la Nube (La Niebla de Silent Hill):

En el archivo application.yml, configuré las variables de entorno de esta manera: url: ${SPRING_DATASOURCE_URL:...}. Esto significa que cuando despliegues tu aplicación y uses una base de datos en la nube (como Supabase, Neon.tech o Railway), solo tendrás que configurar esas variables secretas en tu plataforma de hosting (Render, Heroku, etc.), sin necesidad de cambiar una sola línea de código o exponer tus contraseñas en GitHub. El sistema buscará esas variables mágicas, y si no las encuentra, usará el localhost por defecto.
