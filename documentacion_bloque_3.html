Reporte de Misión: Bloque 3 - Gestión de Cursos (CRUD)

En este bloque programamos la lógica para que el administrador pueda crear, editar y ocultar cursos, manteniendo un alto estándar de código limpio y validaciones estrictas.

1. Borrado Lógico (Soft Delete) vs Físico

Decidimos no borrar los cursos de la base de datos por temas de auditoría, así que añadimos la variable private Boolean isActive; en CourseEntity.java.

  • Borrado Físico (Hard Delete): Es lanzar un comando DELETE FROM real. La fila desaparece de la base de datos para siempre. Si borras por accidente, necesitas un backup (respaldo).
  • Borrado Lógico (Soft Delete): Bajo el capó, es un simple comando UPDATE. Cuando el admin da clic en "Borrar", en realidad el sistema cambia la variable a isActive = false. Luego programamos el controlador de la Vista Pública para que solo muestre cursos con isActive = true, dando la ilusión de que se borró, aunque siga ahí para el administrador.

2. DTO (Data Transfer Object) y Validaciones Automáticas

En el archivo CourseRequest.java:

public class CourseRequest {
    @NotBlank(message = "El título es obligatorio")
    private String title;
}
  • DTO (Data Transfer Object): Es un objeto "mensajero". No es una tabla de base de datos (Entidad) ni es el núcleo del negocio (Modelo). Es solo una mochila temporal para transportar la información (JSON) que el usuario manda desde su navegador web hasta el Controlador de Java.
  • @NotBlank y @Valid: Son anotaciones del paquete jakarta.validation. Actúan como guardias en la puerta del Controlador. Si un usuario intenta enviar un Curso pero deja el título vacío, la anotación intercepta el intento y arroja automáticamente un error HTTP 400 (Bad Request), sin molestar a la base de datos.
Analogía de League of Legends: Un DTO es como un campeón aliado usando la habilidad "Recall" (Vuelta a base) para llevar oro. Es un transporte seguro de un punto A hacia un punto B. El @Valid funciona exactamente igual que el rayo láser de la Fuente (la zona de inicio de la base): si un enemigo camuflado (datos corruptos o vacíos) intenta colarse usando el transporte de tu aliado, el láser de la Fuente lo detecta y lo calcina al instante antes de que pueda tocar el interior de tu base (tu lógica de negocio).

3. Manejo Inteligente de Nulos (Optionals)

En el archivo AdminCourseUseCase.java usamos esto para buscar cursos por ID:

Course existingCourse = courseRepositoryPort.findById(id)
    .orElseThrow(() -> new IllegalArgumentException("Curso no encontrado"));
  • Optional: La función findById de Spring no devuelve el curso pelado; devuelve una "Caja de Schrödinger" llamada Optional. Esa caja puede tener el curso adentro, o puede estar totalmente vacía si el ID no existe en la base de datos. Esto previene el infame error NullPointerException que hace crashear las aplicaciones Java.
  • .orElseThrow(): Le damos una instrucción a la caja: "Intenta sacar el curso de tu interior; pero si por casualidad estás vacía, en lugar de romper todo el sistema de forma caótica, lanza una pequeña excepción (IllegalArgumentException) controlada con un mensaje amigable".
