Reporte de Misión: Bloque 4 - Gestión de Categorías (CRUD)

Hemos implementado la infraestructura para gestionar libremente las Categorías del sistema y hemos aplicado reglas de integridad referencial estrictas para evitar que los datos se corrompan si el usuario comete un error.

1. Borrado Físico (Hard Delete)

[Satisface HU-012] Eliminar categoría

A diferencia de los cursos (que se ocultaban con isActive), para las categorías aplicamos un borrado físico y letal en CategoryPersistenceAdapter.java.

public void deleteById(Long id) {
    jpaCategoryRepository.deleteById(id);
}
  • deleteById: Este método nativo de Spring Data JPA envía un comando SQL literal a la base de datos: DELETE FROM categories WHERE id = X. No deja rastro.

2. Seguro de Vida: Integridad Referencial

[Satisface HU-010 y HU-011] Creación/Edición correcta y [Satisface HU-012] Impedir eliminación si hay cursos

Si borráramos una Categoría que tiene cursos adentro de forma física, los cursos se quedarían flotando en el limbo buscando un "padre" que ya no existe (corrupción de datos u orfandad). Para evitarlo, pusimos un seguro en AdminCategoryUseCase.java.

List<Course> courses = courseRepositoryPort.findByCategoryId(id);
if (!courses.isEmpty()) {
    throw new IllegalStateException("No se puede eliminar la categoría...");
}
  • courses.isEmpty(): El sistema le pregunta a la base de datos: "Tráeme la lista de todos los cursos que estén amarrados a la categoría que estoy a punto de borrar". Si la lista devuelta no está vacía, el código entra al condicional if y aborta agresivamente la operación lanzando un IllegalStateException.
Analogía de League of Legends: Las reglas estrictas de borrado que creamos son como la mecánica de inmunidad de las estructuras. Imagina que el Inhibidor enemigo (La Categoría) quiere ser destruido por tu equipo (El Administrador). El sistema del juego no te va a dejar hacerle daño al Inhibidor si aún está en pie la Torreta que lo defiende (Los cursos asignados a esa categoría). El administrador se verá forzado a derribar la Torreta primero (eliminar los cursos o reasignarlos a otra categoría) antes de poder atacar al Inhibidor.

3. Manejo Elegante de Errores en el Controlador

¿Qué pasa con ese error que arrojamos arriba? Lo capturamos en AdminCategoryController.java para que no crashee la página.

@DeleteMapping("/{id}")
public ResponseEntity<Void> deleteCategory(@PathVariable Long id) {
    try {
        adminCategoryUseCase.deleteCategory(id);
        return ResponseEntity.noContent().build();
    } catch (IllegalStateException e) {
        return ResponseEntity.status(HttpStatus.CONFLICT).build();
    }
}
  • try-catch: El controlador intenta (try) ejecutar el borrado normal. Si el Caso de Uso (UseCase) estalla de coraje y lanza el IllegalStateException, el controlador atrapa (catch) la bomba antes de que destruya el servidor.
  • HttpStatus.CONFLICT (409): En lugar de enviar un fatal error 500 al front-end, envía un elegante código 409 Conflict. Esto es el estándar web para decir: "Entendí tu petición, pero choca (entra en conflicto) con las reglas actuales de mis datos".
