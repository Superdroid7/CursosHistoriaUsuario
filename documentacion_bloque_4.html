Reporte de Misión: Bloque 4 - Gestión de Categorías (CRUD)

Hemos implementado la infraestructura para gestionar libremente las Categorías del sistema y hemos aplicado reglas de integridad referencial estrictas para evitar que los datos se corrompan.

1. Arquitectura del CRUD (Categorías)

  • Hard Delete (Borrado Físico): A diferencia de los cursos, para las categorías se eligió el borrado permanente real (usando deleteById en JPA).
  • Validación de Integridad Restrictiva: En el Caso de Uso (AdminCategoryUseCase) implementamos una regla estricta: antes de intentar borrar una categoría, hacemos una consulta para ver si hay cursos amarrados a ella. Si los hay, lanzamos una excepción IllegalStateException y bloqueamos la transacción devolviendo un código HTTP 409 Conflict.
  • El DTO (CategoryRequest): Validamos que no nos puedan enviar el nombre en blanco usando la validación nativa @NotBlank.

2. Bloques de Código Clave

Analogía de League of Legends: Las reglas estrictas de borrado que creamos son como las Torres de Nexo. Imagina que el Nexo (La Categoría) quiere ser destruido por el equipo enemigo (El Administrador intentando borrar). El sistema no le va a dejar atacar al Nexo si aún están en pie las Torres del Nexo (Los cursos asignados a esa categoría). El administrador se verá forzado a eliminar los cursos primero o reasignarlos antes de poder derribar la categoría.

3. Anotaciones Clave Utilizadas

  • @PostMapping / @PutMapping: Reciben y mutan los datos.
  • ResponseEntity.status(HttpStatus.CONFLICT): Es la forma elegante en que nuestro controlador REST le dice a un cliente web (React/Angular): "Hay un choque de información (conflicto), no puedo borrar esta categoría".
