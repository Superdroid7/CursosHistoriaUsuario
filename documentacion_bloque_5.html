Reporte de Misión: Bloque 5 - Gestión de Imágenes Físicas

Hemos habilitado el sistema de archivos del servidor para almacenar imágenes reales. Esto requirió interactuar con el sistema operativo (Windows/Linux) usando la librería Java NIO (New I/O).

1. Anatomía de la Ubicación de Archivos (Path)

En el archivo FileSystemStorageAdapter.java necesitamos crear la carpeta "uploads" de forma automática si no existe.

Path fileStorageLocation = Paths.get("uploads").toAbsolutePath().normalize();
Files.createDirectories(fileStorageLocation);
  • ¿Qué es la clase Path? Hasta ahora, para manejar textos usábamos String. Pero un Path no es una simple cadena de texto; es un objeto avanzado y especializado que representa una "ruta" literal dentro del disco duro de tu computadora (ej: C:/Users/.../uploads). El objeto Path sabe cómo navegar por directorios, cómo devolverse hacia atrás y cómo leer los permisos del disco.
  • Paths.get("uploads"): Le dice a Java que apunte o localice la carpeta llamada "uploads" en la raíz del proyecto.
  • Files.createDirectories: Es la orden directa que ejecuta Java para decirle al Sistema Operativo de tu PC que cree esa carpeta físicamente si aún no existe.

2. Recepción y Sobreescritura de Archivos

public String storeFile(MultipartFile file) {
    String fileName = UUID.randomUUID().toString() + "_" + originalName;
    Path targetLocation = this.fileStorageLocation.resolve(fileName);
    Files.copy(file.getInputStream(), targetLocation, StandardCopyOption.REPLACE_EXISTING);
    ...
}
  • MultipartFile: Cuando envías un JSON, viaja por internet como un texto simple. Pero cuando envías una foto, esta viaja troceada en pedacitos binarios. MultipartFile es la clase mágica de Spring capaz de atrapar esos pedazos desde la red y reensamblar la imagen original.
  • UUID: Si dos administradores distintos suben una foto llamada perrito.png, la segunda sobreescribiría y destruiría a la primera en la carpeta. Para evitar este caos, usamos la clase UUID (Identificador Único Universal), que genera un código alfanumérico gigante e irrepetible (ej: 550e8400-e29b_perrito.png) para que cada archivo tenga un nombre único mundialmente.
  • Files.copy: Toma el flujo de datos líquidos de la imagen (InputStream) y los copia físicamente en la ubicación que designamos en el disco duro.
Analogía de Resident Evil: Imagina que el MultipartFile es una Hierba Verde que recoges en el suelo de la Mansión Spencer. Mientras la tienes en la mano, es algo temporal y volátil. El Path es saber exactamente el mapa, en qué sala y en qué piso está la caja segura (el Baúl de inventario). La clase UUID es ponerle una etiqueta única a tu hierba para no confundirla con las otras millones que guardes. Y finalmente, Files.copy es la animación física de abrir el baúl y depositar la hierba adentro para resguardarla permanentemente.

3. Exposición de Recursos (WebMvcConfigurer)

Por temas de extrema seguridad, Spring Boot actúa como un búnker cerrado y bloquea el acceso a todas las carpetas internas de tu proyecto. En WebConfig.java abrimos un pequeño conducto de ventilación seguro.

registry.addResourceHandler("/uploads/**")
        .addResourceLocations("file:" + uploadAbsolutePath + "/");
  • ResourceHandlerRegistry: Es un sistema de mapeo y enrutamiento interno. Aquí le damos una instrucción precisa al servidor web: "Si un navegador de internet te solicita una URL que empiece por /uploads/ (ejemplo: localhost:8080/uploads/foto.png), no intentes buscar código Java en un Controlador. En su lugar, ve directamente a buscar el archivo físico en el disco duro (file:) y envíaselo al usuario como una imagen".
