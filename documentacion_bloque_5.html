Reporte de Misión: Bloque 5 - Gestión de Imágenes Físicas

Hemos habilitado el sistema de archivos del servidor para almacenar imágenes reales. Esto completa la HU-013 permitiendo mejorar la estética visual de la landing page.

1. Arquitectura de Almacenamiento (I/O)

  • Separación de Responsabilidades: Fieles a la arquitectura limpia, creamos FileStoragePort en la capa de Dominio, de manera que la lógica de negocio solo sabe que "necesita guardar un archivo". La implementación técnica recae en FileSystemStorageAdapter en la capa de Infraestructura.
  • Rutas Estáticas: Spring Boot por defecto no muestra carpetas internas del disco duro. Tuvimos que usar un configurador de MVC (WebConfig.java) para mapear virtualmente el tráfico de /uploads/** a la carpeta física del proyecto.
  • Seguridad de Archivos: Para evitar colisiones (dos fotos llamadas igual) o ataques, el sistema renombra el archivo entrante agregándole un identificador único global (UUID).

2. Bloques de Código Clave

Analogía de Resident Evil: El servidor es la Mansión Spencer. Antes, guardábamos información (Cursos) en un cuaderno (Base de datos). Ahora, hemos creado un Baúl de Inventario Físico (la carpeta `uploads/`). Cuando subes una foto, la guardas en el baúl y anotas en tu cuaderno la combinación para encontrarla (la URL de la imagen).

3. Anotaciones Clave Utilizadas

  • consumes = "multipart/form-data": Le dice explícitamente a Spring que este endpoint no va a recibir texto JSON convencional, sino "pedazos" binarios (una imagen).
  • @RequestParam("file"): Extrae el archivo binario enviado en el formulario bajo el campo llamado "file".
